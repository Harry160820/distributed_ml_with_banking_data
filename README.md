## Project Overview

This project simulates a real-world banking data platform using distributed computing. It demonstrates how modern banks process large-scale customer data — from raw storage to predictive analytics and real-time fraud detection.

The pipeline covers five core areas:

| Area | Tool | Purpose |
|------|------|---------|
| Data storage & querying | Hadoop HDFS + Hive | Store bank data and run SQL analytics |
| Exploratory analysis | Apache Spark (PySpark) | Uncover trends, patterns, and anomalies |
| Predictive modeling | Spark ML | Predict term deposit subscription |
| Real-time processing | Spark Structured Streaming | Detect fraudulent transactions live |
| Data parallelism | Spark (distributed execution) | Efficiently process large datasets |

## Step-by-Step Guide

### Step 1 — Bootstrap the project

Run once from your WSL terminal. Creates all directories and installs Python dependencies.

```bash
cd ~
bash banking_ml_project/setup.sh
```

Then verify `config.py` sees all paths correctly:

```bash
python ~/banking_ml_project/config.py
# Expected: [OK] bank.csv found at .../data/bank.csv
```

**What `config.py` does:** It resolves all paths relative to its own location using `Path(__file__).resolve().parent`. Every script imports it with `from config import PATHS, verify_paths`. This means the project works regardless of which directory you `cd` into before running a script.

---

### Step 2 — Prepare data

Place the source dataset:

```bash
cp /path/to/your/bank.csv ~/banking_ml_project/data/bank.csv
```

Generate synthetic transaction data for the streaming job:

```bash
python ~/banking_ml_project/scripts/06_gen_transactions.py
# Output: data/transactions_sim.csv  (2000 rows with realistic amount distribution)
```

---

### Step 3 — Upload to HDFS

Start Hadoop first, then upload:

```bash
# Start Hadoop (if not already running)
start-all.sh

# Verify it's running
jps
# Should show: NameNode, DataNode, ResourceManager, NodeManager

# Upload both files
bash ~/banking_ml_project/scripts/01_upload_to_hdfs.sh
```

This creates two HDFS directories:
- `/banking/raw/bank/` — contains `bank.csv`
- `/banking/raw/transactions/` — contains `transactions_sim.csv`

---

### Step 4 — Create Hive tables

```bash
hive -f ~/banking_ml_project/scripts/02_hive_setup.hql
```

Creates `banking_db.bank_customers` as an **external table** pointing to `/banking/raw/bank/` in HDFS. Hive does not copy the data — it reads it in place. After completion, the script runs verification queries showing row count and subscription rate breakdown.

---

### Step 5 — Spark EDA

```bash
spark-submit ~/banking_ml_project/scripts/03_spark_eda.py \
  2> ~/banking_ml_project/logs/eda.log
```

Generates the following in `artifacts/eda/`:

| Output file | Description |
|-------------|-------------|
| `age_distribution.png` | Histogram of customer ages |
| `balance_by_subscription.png` | Balance distribution split by target variable |
| `correlation_heatmap.png` | Pearson correlation across all numeric features |
| `job_subscription.csv` | Subscription rate grouped by job type |
| `summary_stats.csv` | Count, mean, stddev, min, max for numeric columns |

---

### Step 6 — Spark ML

```bash
spark-submit ~/banking_ml_project/scripts/04_spark_ml.py \
  2> ~/banking_ml_project/logs/ml.log
```

Trains a **Random Forest classifier** (100 trees, maxDepth=6) using a full Spark ML Pipeline:

```
StringIndexer → OneHotEncoder → VectorAssembler → StandardScaler → RandomForestClassifier
```

Outputs:

| Output | Location |
|--------|----------|
| Trained model | `artifacts/models/rf_model/` |
| Evaluation report | `artifacts/reports/ml_report.txt` |

The report contains AUC-ROC, Accuracy, and F1 Score on the 20% held-out test set.

---

### Step 7 — Spark Streaming

```bash
spark-submit ~/banking_ml_project/scripts/05_spark_streaming.py \
  2> ~/banking_ml_project/logs/streaming.log
```

Watches `data/stream_input/` for new CSV files. When a file appears, it:
1. Parses each transaction row
2. Computes a `fraud_score` based on amount thresholds
3. Filters rows where `fraud_score >= 0.8` or `is_fraud_flag == 1`
4. Appends flagged rows to `artifacts/streaming/alerts/`

**To feed a new batch while streaming is running:**

```bash
# In a second terminal:
cp ~/banking_ml_project/data/transactions_sim.csv \
   ~/banking_ml_project/data/stream_input/batch_02.csv
```

Stop the streaming job with `Ctrl+C` when done.

## Configuration System

`config.py` is the single source of truth for all file paths. Import it at the top of any new script you add:

```python
import sys
from pathlib import Path

sys.path.insert(0, str(Path(__file__).parent.parent))
from config import PATHS, verify_paths

verify_paths()  # creates missing dirs, warns if bank.csv absent

# Use paths like:
df = spark.read.csv(str(PATHS["bank_csv"]), header=True, sep=";")
fig.savefig(PATHS["eda"] / "my_plot.png")
```

Available keys in `PATHS`:

| Key | Resolves to |
|-----|-------------|
| `root` | Project root directory |
| `data` | `data/` folder |
| `bank_csv` | `data/bank.csv` |
| `transactions_csv` | `data/transactions_sim.csv` |
| `artifacts` | `artifacts/` folder |

| `models` | `artifacts/models/` |
| `streaming` | `artifacts/streaming/` |

## WSL-Specific Notes

**Matplotlib headless mode** — All plot scripts use `matplotlib.use("Agg")` before importing `pyplot`. This prevents crashes when there is no display attached (standard in WSL without an X server).

**Kafka not required** — The streaming job uses Spark's file-source mode instead of Kafka. Drop any CSV with the transaction schema into `data/stream_input/` to trigger processing. This avoids the complex Kafka broker setup that is fragile in WSL.

**Hadoop startup** — Always run `start-all.sh` and wait ~30 seconds before running HDFS or Hive steps. Check with `jps` that all four daemons are running.

**JAVA_HOME** — If Hadoop or Spark fails to start, ensure `JAVA_HOME` is set in your `~/.bashrc`:

```bash
export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64
export PATH=$JAVA_HOME/bin:$PATH
```

**Spark PATH** — If `spark-submit` is not found:

```bash
export SPARK_HOME=/opt/spark   # adjust to your install path
export PATH=$SPARK_HOME/bin:$PATH
```

## Troubleshooting

| Problem | Likely cause | Fix |
|---------|-------------|-----|
| `hdfs: command not found` | Hadoop not in PATH | Add `$HADOOP_HOME/bin` to PATH |
| `hive: command not found` | Hive not in PATH | Add `$HIVE_HOME/bin` to PATH |
| `[WARNING] bank.csv missing` | File not placed | `cp your_file ~/banking_ml_project/data/bank.csv` |
| HDFS upload fails | Hadoop not running | Run `start-all.sh`, then `jps` to verify |
| Hive query hangs | Metastore not initialized | Run `schematool -initSchema -dbType derby` |
| Spark OOM error | Insufficient driver memory | Increase `spark.driver.memory` in the script (default: 2–3g) |
| Streaming: no alerts produced | Batch not in stream_input | Copy a CSV to `data/stream_input/` manually |
| Plots not saving | artifacts/eda/ missing | Run `python config.py` to recreate directories |

## Project Objectives

This project fulfills the following academic and practical objectives:

1. **Data storage and management** — Hadoop HDFS stores raw CSVs; Hive provides an SQL interface for querying without moving data.

2. **Exploratory Data Analysis** — PySpark reads the dataset in a distributed context, computes aggregations, and produces visualizations saved to the artifacts folder.

3. **Predictive modeling** — A Spark ML Pipeline encodes features, scales them, and trains a Random Forest classifier to predict whether a customer will subscribe to a term deposit.

4. **Real-time transaction analysis** — Spark Structured Streaming watches a directory for incoming transaction files and continuously filters high-risk transactions into an alerts store.

5. **Data parallelism** — All Spark jobs use `local[*]` mode, utilizing all available CPU cores. In a cluster deployment, the same scripts work unchanged with a `spark://master:7077` master URL.
